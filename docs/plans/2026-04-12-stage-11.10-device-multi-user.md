# Stage 11.10 — 디바이스 멀티유저 (머신 단위 디바이스)

**Date**: 2026-04-12
**Branch**: `feat/hosted-migration`
**Depends on**: 11.9 (device auto-claim)

## Why

현재 `devices.user_id` 구조는 하나의 러너(머신)를 하나의 Google 계정만 사용할 수 있다.
공용 컴퓨터나 같은 컴퓨터에서 계정을 바꿔 쓰는 경우 `device_offline`이 뜨고 러너를 다시 페어링해야 한다.

디바이스는 머신(러너) 단위이고, 여러 유저가 같은 디바이스를 인가받을 수 있어야 한다.

## 목표

- `devices` 테이블에서 `user_id` FK 제거
- `device_users(device_id, user_id)` 중간 테이블로 다대다 관계 표현
- 러너는 이미 페어링된 상태에서도 startup 시 항상 claim을 등록 → 새 유저가 동일 머신에서 Connect 클릭 시 기존 디바이스에 자신을 추가하는 것으로 처리
- 웹/백엔드/러너 동작은 유저 입장에서 기존과 동일하게 유지

## 스키마 변경

### device_users 테이블 (신규)

```ts
export const device_users = pgTable("device_users", {
  device_id: uuid("device_id")
    .notNull()
    .references(() => devices.id, { onDelete: "cascade" }),
  user_id: uuid("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  created_at: timestamp("created_at").notNull().default(sql`now()`),
}, (t) => ({
  pk: primaryKey({ columns: [t.device_id, t.user_id] }),
}));
```

### devices 테이블 변경

- `user_id` 컬럼 및 FK 제거
- 나머지 컬럼 유지

### 마이그레이션 순서 (한 파일로 처리)

```sql
-- 1. 중간 테이블 생성
CREATE TABLE "device_users" (
  "device_id" uuid NOT NULL REFERENCES "devices"("id") ON DELETE CASCADE,
  "user_id"   uuid NOT NULL REFERENCES "users"("id")   ON DELETE CASCADE,
  "created_at" timestamp NOT NULL DEFAULT now(),
  PRIMARY KEY ("device_id", "user_id")
);

-- 2. 기존 devices.user_id 데이터 이전
INSERT INTO "device_users" ("device_id", "user_id")
SELECT "id", "user_id" FROM "devices"
WHERE "user_id" IS NOT NULL;

-- 3. devices.user_id 컬럼 삭제
ALTER TABLE "devices" DROP COLUMN "user_id";
```

`drizzle-kit generate`로 생성하되, 데이터 이전 구문(2번)은 수동으로 삽입한다.
마이그레이션 번호: `0002`.

## 백엔드 변경

### pairing.ts

**insertDevice**: `userId` 파라미터 유지, 하지만 `devices` INSERT에서 제거하고 `device_users` INSERT 추가.

```ts
await db.insert(devices).values({ id, label, hostname, os, runnerVersion, workspaceRoot, tokenHash });
await db.insert(device_users).values({ device_id: id, user_id: userId });
```

**listDevices** (`GET /api/devices`): `device_users` JOIN으로 변경.

```ts
db.select({ ...deviceFields })
  .from(devices)
  .innerJoin(device_users, eq(device_users.device_id, devices.id))
  .where(and(eq(device_users.user_id, userId), isNull(devices.revoked_at)));
```

**revokeDevice** (`POST /api/devices/:id/revoke`): device_users 통해 소유권 검증.

```ts
const membership = await db.select()
  .from(device_users)
  .where(and(eq(device_users.device_id, id), eq(device_users.user_id, userId)))
  .limit(1);
if (membership.length === 0) { reply.code(403); return; }
await db.update(devices).set({ revoked_at: new Date() }).where(eq(devices.id, id));
```

**approveEntry** (auto-claim approve 내부): 신규/기존 디바이스 분기.

```ts
// claim에 deviceId가 있으면 기존 디바이스에 유저 추가
if (entry.deviceId) {
  await db.insert(device_users)
    .values({ device_id: entry.deviceId, user_id: userId })
    .onConflictDoNothing();
  // 토큰 재발급 없음 — 기존 토큰 그대로
  return { status: "authorized", deviceId: entry.deviceId };
}
// 없으면 신규 디바이스 생성 (기존 로직)
```

### rpc.ts — listActiveDeviceIds

### runnerSocket.ts — findByTokenHash

`devices.user_id` SELECT 제거. device_users JOIN으로 userId 조회:

```ts
const rows = await db
  .select({ id: devices.id })
  .from(devices)
  .where(and(eq(devices.token_hash, tokenHash), isNull(devices.revoked_at)))
  .limit(1);

if (rows.length === 0) return null;
const deviceId = rows[0].id;

// userId 목록 (device_users)
const userRows = await db
  .select({ userId: device_users.user_id })
  .from(device_users)
  .where(eq(device_users.device_id, deviceId));

return { id: deviceId, userIds: userRows.map(r => r.userId) };
```

DeviceHub의 attach/route 로직도 단일 userId → userIds 배열로 대응 조정.

### auth/device-claim 엔드포인트 변경

**POST /auth/device-claim** (러너가 호출): body에 `deviceId?: string` 추가.
러너가 이미 페어링된 경우 자신의 device ID를 함께 보낸다.

**POST /api/device-claim/approve** 응답 추가:
- `{ status: "authorized", deviceId }` — 기존 디바이스에 유저 추가된 경우

웹 `ConnectConsentModal`의 폴링 로직: `approved` / `authorized` 둘 다 성공으로 처리.

## 러너 변경

### autoClaim.ts

항상 startup 시 claim 등록. 이미 토큰이 있으면 deviceId도 함께 전송:

```ts
export async function registerClaim(opts: {
  backendUrl: string;
  deviceId?: string;   // 이미 페어링된 경우
  abortSignal?: AbortSignal;
}): Promise<{ claimId: string; pollToken: string; expiresAt: string }>;
```

### index.ts

```ts
const existingToken = await loadDeviceToken();
const existingDeviceId = existingToken ? await resolveDeviceId(existingToken) : undefined;

// 항상 claim 등록 (신규 유저가 이 머신에 접근할 수 있도록)
const claim = await registerClaim({ backendUrl, deviceId: existingDeviceId });

if (!existingToken) {
  // 새 디바이스: 폴링해서 토큰 받아서 저장
  const result = await pollClaim(claim);
  await saveDeviceToken(result.token);
} else {
  // 기존 디바이스: 폴링하되 "authorized" 응답은 토큰 변경 없이 무시
  void pollClaimNonBlocking(claim).catch(() => {});
}

await mainHosted();
```

`resolveDeviceId`: 토큰으로 device ID를 찾는 헬퍼 (로컬 저장 또는 백엔드 조회).

## 응답 타입 변경 (web client.ts)

type ApproveDeviceClaimResult =

`ConnectConsentModal`에서 `approved` | `authorized` 둘 다 폴링 진입으로 처리.
