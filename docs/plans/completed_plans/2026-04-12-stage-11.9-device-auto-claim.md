# Stage 11.9 — Device auto-claim (zero-friction pairing)

**Date**: 2026-04-12
**Branch**: `feat/hosted-migration`
**Depends on**: 11.6 (dev automation landed), 11.8 (profile parity)

## Why

The current pairing flow is unusable for a hosted service:

1. Web UI asks for `Device label` and `Workspace root path`. End users on Mac/Windows cannot be asked to paste an absolute filesystem path.
2. `devices.workspace_root NOT NULL` is load-bearing in the schema, but the runner **ignores it entirely** — `runnerContext.ts:44` uses `os.homedir()` regardless of what the web form captured. The backend value is cosmetic debt.
3. User must manually: start runner with `JASOJEON_MODE=pair`, read a code off the web UI, paste it into their terminal. Three terminals of friction per device.

This stage removes all of that. After logging in, the user sees a consent modal; on consent, the runner already running on their machine is auto-paired without any further interaction.

## Goal

**Clean-slate flow**:
1. User opens the hosted web UI and logs in with Google.
2. First-time (or unpaired) bootstrap hits the BootstrapGate → `device_offline` branch.
3. Consent modal: *"Jasojeon uses a local CLI runner to read and write files on your machine. Do you consent?"* → single **Connect** button.
4. On Connect: backend looks for a pending device-claim from the user's source IP within the last 2 minutes. If exactly one match → auto-approves → returns a device token → the runner's polling loop picks up the token on its next poll → runner saves it and initiates the outbound WS connection → web sees the device online and hides the modal. Total elapsed: ≤ 3 seconds.
5. If no claim matches: modal shows *"Runner not detected. Start the runner on your machine, then click Retry."* with a copy-paste install hint (no hardcoded installer yet — pointer text only; actual installer is 11.6.B).

The runner itself, on startup, if unpaired, begins registering a pending claim **automatically** and polls for approval. No `JASOJEON_MODE=pair`, no environment variable, no code entry.

## Scope

### In scope

### 2. Backend — `packages/backend/src/routes/pairing.ts` rewrite

**Remove** (delete code entirely, not deprecate):
- `POST /api/pairing/start`
- `POST /api/pairing/claim`
- `StartBodySchema`, `ClaimBodySchema`, `PairingValue`, rate-limit logic
- `generatePairingCode`, `normalizeCode`

**Keep**:
- `GET /api/devices`
- `POST /api/devices/:id/revoke`
- `DeviceStore` / `DrizzleDeviceStore` — but update `insertDevice` signature to take `{ id, userId, label, hostname, os, runnerVersion, workspaceRoot, tokenHash }` where `workspaceRoot` is optional.

**Add**:
- `POST /auth/device-claim` (unauthenticated — runner calls this)
  - Body: `{ hostname: string, os: string, runnerVersion: string, workspaceRoot?: string }` (all validated via zod, `hostname`/`os` max 200 chars).
  - Response: `{ claimId: string, pollToken: string, expiresAt: string }`.
  - Side effect: stores claim in Redis under `claim:<claimId>` with TTL 600s. Value includes client IP (from `request.ip` — trust Fastify's proxied IP since we terminate TLS behind nginx in prod; in dev it's just the connecting IP).
  - Rate limit: max 10 claims / minute per IP (Redis counter).

- `GET /auth/device-claim/:claimId?pollToken=xxx` (unauthenticated)
  - Validates `pollToken` matches the stored one.
  - Returns one of:
    - `{ status: "pending" }`
    - `{ status: "approved", token: string, deviceId: string, userId: string }` — **consumes the claim**, deletes Redis key.
    - `{ status: "rejected" }` — user explicitly declined.
    - `{ status: "expired" }` — TTL lapsed.
  - Long-poll friendly: cap response time at 25s; if still pending, return `pending` so the runner re-polls. Use `setTimeout` + Redis `XREAD`-style wait OR simple polling inside the handler (prefer the latter for simplicity — 500ms internal tick, 25s max).

- `POST /api/device-claim/approve` (session required)
  - Body: `{ claimId?: string }` — if omitted, backend looks for exactly one pending claim matching `request.ip`.
  - Returns:
    - `{ status: "approved", deviceId, label }` — single match found, claim approved, device row inserted, token hash stored.
    - `{ status: "no_claim" }` — no pending claim from this IP in the last 2 minutes.
    - `{ status: "multiple_claims", claims: [{ claimId, hostname, os }...] }` — caller must retry with explicit `claimId`.
  - On approval, backend generates the device token (32 random bytes hex), stores `sha256(token)` in `devices.token_hash`, updates the Redis claim entry with `{ status: "approved", token, deviceId, userId }`.

**Redis key layout**:
- `claim:<claimId>` — JSON `{ hostname, os, runnerVersion, workspaceRoot, ip, pollToken, status: "pending"|"approved"|"rejected", token?, deviceId?, userId? }`, TTL 600s.
- `claim-rate:<ip>` — rate limit counter, TTL 60s.

### 3. Runner — `packages/runner/src/hosted/pairingClient.ts` rewrite

Delete `claimPairingCode`. Replace with:

```ts
export async function autoClaimDevice(opts: {
  backendUrl: string;
  abortSignal?: AbortSignal;
}): Promise<{ token: string; deviceId: string; userId: string }>;
```

Flow:
1. POST `/auth/device-claim` with `{ hostname: os.hostname(), os: process.platform, runnerVersion: <from package.json> }`. Receive `{ claimId, pollToken, expiresAt }`.
2. Log to stdout: `[runner] Waiting for approval at ${backendUrl} (claim ${claimId.slice(0, 8)}...)`
3. Loop: GET `/auth/device-claim/:claimId?pollToken=...`. 1s delay between polls. Max wait = 10 minutes (same as claim TTL). Honor `abortSignal`.
4. On `approved` → return `{ token, deviceId, userId }`.
5. On `rejected` / `expired` / any network error after 3 retries → throw a typed error.

Runner version: read `packages/runner/package.json` at build time (static import) or resolve at runtime — prefer runtime via `readFileSync` at module load, cached.

### 4. Runner — `packages/runner/src/index.ts` integration

Remove the `JASOJEON_MODE === "pair"` branch entirely. On boot:
- If `loadDeviceToken()` returns a token → proceed to `mainHosted()` as today.
- If no token → call `autoClaimDevice({ backendUrl: JASOJEON_BACKEND_URL })`, save result via `saveDeviceToken()`, then fall through to `mainHosted()`. No second process invocation.

Log messages should make it obvious to anyone tailing the runner log what's happening.

## Judgment latitude

- If the drizzle-kit generator emits migration SQL that differs from the hand-written example above, trust the generator.
- If Fastify's `request.ip` doesn't give a usable value under the Vite proxy (proxy is on same host anyway), fall back to `request.socket.remoteAddress`.
- If long-poll timeout (25s) fights with Fastify's default idle timeout, use short 1s polls instead — correctness > latency.
- If `@fastify/oauth2` session paths conflict with the new `/auth/device-claim` route, move device-claim to `/api/device-claim/...` to keep `/auth/*` reserved for OAuth.
