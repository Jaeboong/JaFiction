# 프로젝트 인사이트 모달 뷰어 설계서

> 인사이트 생성 완료 후 ProjectsPage에서 "인사이트 보기" 버튼으로 전환하고, 모달에서 좌측 탭/우측 본문 구조로 인사이트를 읽도록 만드는 설계

---

## 1. 문제 정의

현재 인사이트 생성 플로우는 러너 쪽 로직은 충분히 준비돼 있지만, 웹 UI는 이를 거의 소비하지 못한다.

현재 상태

- `POST /api/projects/:projectSlug/insights/generate`는 인사이트 문서를 생성한다
- `GET /api/projects/:projectSlug/insights`는 인사이트 워크스페이스 상태를 제공한다
- 생성 문서는 저장소에 `company-insight.md`, `job-insight.md`, `application-strategy.md`, `question-analysis.md`로 저장된다

하지만 ProjectsPage에서는 지금 아래만 보인다.

- `공고 분석` 버튼
- `인사이트 생성` 버튼

즉, 사용자는 인사이트를 만든 뒤에도

- 어디서 결과를 보는지 알기 어렵고
- 생성된 문서가 컨텍스트 문서 목록에 섞여 있거나 보이지 않으며
- 인사이트가 독립 산출물이라는 느낌을 받지 못한다

사용자 요구는 명확하다.

- 생성 후 버튼은 `인사이트 생성` 대신 `인사이트 보기`
- 클릭 시 모달 오픈
- 모달은 좌측 탭, 우측 본문 구조
- 탭 예시: `기업 분석`, `직무 분석` 등

---

## 2. 목표

- 인사이트가 생성되면 ProjectsPage에서 즉시 "볼 수 있는 결과물"로 승격한다
- 인사이트는 일반 컨텍스트 문서 목록과 분리된 전용 UX로 표시한다
- 좌측 탭 / 우측 본문 구조로 빠르게 탐색할 수 있게 한다
- 생성된 인사이트 문서는 저장소에는 그대로 남겨 런타임 컨텍스트 재사용성을 유지한다

---

## 3. 범위와 가정

### 3.1 범위

- 대상 화면은 `ProjectsPage > ProjectWorkspace`
- 대상 데이터는 기존 `GET /api/projects/:projectSlug/insights`
- 대상 문서는 아래 4개 고정 인사이트 아티팩트
  - 기업 분석
  - 직무 분석
  - 지원 전략
  - 문항 분석

### 3.2 가정

- 인사이트 문서는 저장소 내부적으로는 project documents로 남아도 된다
- 다만 **일반 컨텍스트 문서 리스트 UX에서는 숨기거나 별도 취급**하는 것이 맞다
- 이번 단계에서는 모달 기반 읽기 경험이 우선이며, 전용 페이지 분리는 하지 않는다

---

## 4. 현재 상태 진단

### 4.1 러너는 이미 인사이트 워크스페이스 데이터를 제공한다

`packages/runner/src/routes/insightsRouter.ts`의 `buildInsightWorkspaceState()`는 이미 아래 구조를 반환한다.

- 프로젝트 메타
- 인사이트 생성 시각
- OpenDART 메타
- source manifest
- 탭형 문서 배열
  - `company`
  - `job`
  - `strategy`
  - `question`

즉, 좌측 탭/우측 본문 UI를 만들기 위한 서버 데이터는 이미 존재한다.

### 4.2 웹 UI가 그 상태를 전혀 쓰지 않는다

ProjectsPage는 생성 요청만 보낼 수 있고, 조회 UX가 없다.

결과적으로 사용자는 "인사이트 생성"을 눌러도

- 완료 여부는 상태 텍스트로만 추정하고
- 본문을 확인할 방법이 없고
- 왜 이 문서가 만들어졌는지 파악하기 어렵다

### 4.3 컨텍스트 문서와 인사이트 문서가 섞이는 인상

저장소 관점에서 generated documents를 project documents로 저장하는 것은 맞다.

이유

- ContextCompiler가 그대로 재사용 가능하다
- 실행 시 자동 포함 문서로 다루기 쉽다

하지만 UI에서 이를 일반 "컨텍스트 문서"와 동일 레벨로 보여주면 사용자 경험은 나빠진다.

따라서 저장 구조와 표시 구조를 분리해야 한다.

---

## 5. 접근 옵션

### 옵션 A. ProjectsPage 안에 인라인 패널로 펼치기

- 프로젝트 상세 페이지에 인사이트 섹션을 직접 추가

장점

- 구현이 단순하다
- 모달 없이 바로 보인다

단점

- 본문이 길어지면 페이지가 과하게 무거워진다
- 프로젝트 기본 정보, 컨텍스트, 인사이트가 한 화면에 섞인다

### 옵션 B. 모달 뷰어 도입

- 버튼 클릭 시 전용 모달
- 좌측 탭, 우측 본문

장점

- 사용자가 "결과 보기" 맥락으로 진입한다
- 인사이트를 일반 문서 목록과 분리하기 쉽다
- 추후 재생성/소스 보기/복사 기능을 붙이기 좋다

단점

- 모달 상태 관리가 추가된다

### 옵션 C. 별도 페이지 또는 상위 탭으로 분리

- ProjectsPage 내부 하위 라우트 또는 독립 탭 도입

장점

- 장기적으로 확장성 높다

단점

- 현재 요구 수준에 비해 과하다
- 이동 맥락이 무거워진다

### 권장안

**옵션 B를 권장한다.**

요구사항이 이미 "모달 띄워서 보게 해"에 가깝고, 인사이트를 일반 컨텍스트 UI에서 분리하려는 목적과도 가장 잘 맞는다.

## 6. 제안 UX

### 6.1 버튼 동작

프로젝트 헤더 우측 액션을 아래 규칙으로 바꾼다.

- `insightStatus !== "ready"`: 기본 CTA는 `인사이트 생성`
- `insightStatus === "ready"`: 기본 CTA는 `인사이트 보기`
- 인사이트가 준비된 상태에서도 보조 버튼으로 `다시 생성`을 둘 수 있다

권장 버튼 구성

- 보조: `공고 분석`
- 기본:
  - 준비 전: `인사이트 생성`
  - 준비 후: `인사이트 보기`
- 준비 후 보조 액션: `다시 생성`

### 6.2 모달 오픈 시점

- `인사이트 보기` 클릭 시 lazy fetch로 `/api/projects/:slug/insights` 호출
- 응답 완료 후 모달 표시
- 로딩 중에는 스켈레톤 또는 로딩 상태 표시

### 6.3 모달 레이아웃

```text
┌────────────────────────────────────────────────────────────┐
│ 인사이트 뷰어                               [닫기]         │
├─────────────────┬──────────────────────────────────────────┤
│ 기업 분석        │ 제목                                     │
│ 직무 분석        │ 생성 시각 / 회사 / 직무 / OpenDART 메타     │
│ 지원 전략        │ ---------------------------------------- │
│ 문항 분석        │ 본문(markdown 표시)                       │
│                 │                                          │
└─────────────────┴──────────────────────────────────────────┘
```

좌측

- 고정 폭 탭 목록
- 사용 가능한 문서만 활성 강조

우측

- 탭 제목
- 생성 시각
- 회사명 / 직무명
- 본문

### 6.4 본문 표시 방식

초기 단계 권장

- 마크다운 문자열을 그대로 읽기 쉬운 본문 레이아웃으로 표시
- 필요하면 간단한 heading/paragraph/list 스타일만 적용

이유

- 별도 복잡한 markdown renderer 의존성 없이도 충분히 읽을 수 있다
- 현재 목적은 "바로 보기" 경험 복원이지 문서 편집기가 아니다

## 7. 컨텍스트 문서 목록과의 관계

이 부분이 사용자 불만의 핵심이다.

### 설계 원칙

- 저장소에서는 generated insight documents를 계속 project documents로 유지
- 하지만 ProjectsPage의 일반 `컨텍스트 문서` 목록에서는 **기본적으로 제외**
- 대신 인사이트 전용 모달에서만 보여준다

### 이유

- 런타임 호환성 유지
- ContextCompiler 로직 유지
- 사용자에게는 "업로드/작성 문서"와 "생성된 인사이트"를 다른 개념으로 분리

### 표시 정책

일반 컨텍스트 목록에는 아래 문서명을 필터링한다.

- `company-insight.md`
- `job-insight.md`
- `application-strategy.md`
- `question-analysis.md`

또는 더 나은 방식으로는 문서 메타에 `generatedBy = "insight"` 같은 구분자를 추가할 수 있다. 다만 초기 단계는 파일명 기반 필터로도 충분하다.

## 9. 세부 동작 규칙

### 9.1 생성 완료 전

- 버튼: `인사이트 생성`
- 클릭 시 현재처럼 generate API 호출
- 생성 중에는 버튼 disabled + pending label

### 9.2 생성 완료 후

- 기본 버튼을 `인사이트 보기`로 치환
- 클릭 시 모달 오픈

### 9.3 재생성

- `인사이트 보기` 모달 내부 또는 헤더 액션에 `다시 생성`
- 재생성 후 모달 내용을 새로고침

### 9.4 일부 문서가 비어 있는 경우

- 좌측 탭은 유지
- 우측에 "아직 생성되지 않았습니다" empty state 표시

---

## 10. 에러 처리

대표 시나리오

- 인사이트 생성 실패
- 인사이트 조회 실패
- 일부 문서만 생성됨

표시 정책

- 생성 실패: 기존 헤더 영역에 배너 또는 상태 문구
- 조회 실패: 모달 내부 오류 패널
- partial state: 탭은 유지하고 empty state 표시

---

## 11. 테스트 관점

### 11.1 UI 테스트

- `ready` 상태면 헤더 버튼이 `인사이트 보기`로 바뀐다
- 클릭 시 모달이 열린다
- 좌측 탭 클릭에 따라 우측 본문이 바뀐다

### 11.2 통합 테스트

- generate 후 get insights 호출 시 4개 탭 데이터를 받아온다
- 컨텍스트 문서 목록에서 인사이트 문서는 노출되지 않는다

### 11.3 회귀 포인트

- 인사이트 문서가 저장소에는 계속 남아 있어야 한다
- 실행 컨텍스트 컴파일에 필요한 pinned/generated 흐름은 유지돼야 한다

## 13. 결론

이 요구의 핵심은 "인사이트 생성"을 단순 액션이 아니라 **읽을 수 있는 결과물 경험**으로 바꾸는 것이다.

권장 방향은 다음과 같다.

- `ready` 상태에서 `인사이트 생성`을 `인사이트 보기`로 치환
- 클릭 시 전용 모달 오픈
- 좌측 탭 / 우측 본문 구조로 4개 인사이트를 탐색
- 저장소 내부 문서는 유지하되, 일반 컨텍스트 목록에서는 분리 표시

이 설계는 현재 러너가 이미 제공하는 인사이트 워크스페이스 상태를 그대로 활용하면서, 사용자가 가장 답답해하는 "생성했는데 볼 데가 없음" 문제를 직접 해결한다.
