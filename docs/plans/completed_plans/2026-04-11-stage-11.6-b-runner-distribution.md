# Stage 11.6.B — End-user Runner Distribution Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Allow non-developer end users to install and pair a runner with https://jaeboong.shop using a turnkey artifact that reuses the Stage 11.6 pairing workflow without requiring Node, git, or WSL on their machines.

**Architecture:** Produce a single-file runner binary distributed via a curl-install script that configures supervise.mjs equivalents as native services while maintaining outbound-only WSS connectivity and backend-driven update signaling.

**Tech Stack:** Bun single-file build, curl/bash installer, systemd/launchd service units, backend version APIs, Stage 11.6 pairing APIs, TLS-secured WebSocket runner.

---

## 1. Why / Constraints

- Persona — 일반 비개발자: assumes browser literacy, copy/paste, but not CLI or package manager experience; expects GUI cues and minimal terminal exposure confined to copy/pastable commands delivered via onboarding pages.
- Persona — 파워유저: comfortable with terminal, can inspect logs, can run `curl | bash`, and can troubleshoot service restarts; acts as support channel for regular users, so tooling must expose diagnostic toggles.
- OS Targets: Ubuntu/Debian-class Linux and macOS Sonoma/Sequoia native; WSL counts as Linux but only for fallback; Windows native installers deferred because code signing and service management require Stage 12.
- Network expectations: outbound HTTPS + WSS to jaeboong.shop (443) must succeed; inbound ports unavailable (home routers, enterprise firewalls) thus pairing must use polling/long-lived outbound sockets; proxies unpredictable, so CLI must allow `HTTPS_PROXY` override.
- Security boundaries: never ship git or Node; runner binary may bundle dependencies but must store secrets encrypted using OS keychains when possible; no root-level daemons, user-level services only.
- Operational goal: install + login + pair within 3 pasted commands or fewer, total under 2 minutes on clean machine with broadband; allow uninstall script to clean artifacts.
- Stage 11.6 dependency: rely on proven start-dev-runner pairing UX and supervise.mjs semantics; do not fork logic; share manifest by invoking shared JS entrypoint via embedded bun runtime.
- Telemetry: capture minimal install metadata (timestamp, OS, version, hashed machine id) to help support but respect privacy; add opt-in flag for verbose logging to remote support channel.
- Assumed privileges: installers run under standard user accounts; escalate only to modify PATH or install LaunchAgent; never require sudo except optional `systemctl --user` linger enabling instructions.
- Localization: CLI text remains Korean-first but include English fallback (`--lang en`) because power users may set OS language to English; ensure pair page auto-detects locale.
- Data persistence: configs stored at `~/.jasojeon/config.json`, logs under `~/.jasojeon/logs/YYYY-MM-DD/runner.log`; ensures easy support instructions; logs rotated at 10MB.
- Support channel: CLI exposes `support bundle` command compressing latest logs + anonymized config; zipped artifact uploaded via pre-signed URL for support, ensuring direct path to triage issues raised by Stage 11.6.B testers.
- Onboarding copy: marketing site should surface CTA "내 PC에서 러너 실행" linking to instruction page hosting curl command; plan coordinates with docs team to ensure messaging matches CLI output.
- Environmental constraints: corporate laptops may enforce Zscaler-style SSL inspection; CLI must detect certificate mismatch early and prompt user to install CA or use `--ca-bundle` argument referencing IT-provided file.
- Observability: runner heartbeat metrics forwarded to backend to detect offline clients; Stage 11.6.B adds column in admin UI for "Last seen" per user to help support; requires new GraphQL/resolver work.
- Disaster recovery: plan includes `jasojeon-runner factory-reset` removing device token and forcing re-pair; necessary when user sells laptop.

## 3. Recommendation + Reasoning

- MVP deliverable: bun single-file binaries per OS served via CDN + curl installer that downloads, verifies checksum/signature, unpacks to `~/.jasojeon/runner-<version>/`, symlinks `jasojeon-runner` into `~/bin`, and registers background service wrappers.
- Reasons:
  - Single artifact ensures parity between dev automation and prod; bundling Stage 11.6 start-dev-runner pairing logic avoids reimplementation.
  - Binary eliminates Node/npm requirement; bundler reduces cold-start time (<300ms) and includes pinned dependencies, so users avoid `npm install` entirely.
  - Curl installer allows Step-0: copy/paste; Step-1: login via browser; Step-2: pairing auto-handled; matches acceptance criteria.
  - Flow supports offline caching and version pinning via `JASOJEON_RUNNER_VERSION=<tag> curl ...`; CLI can fallback to previous version if handshake fails.
  - Security: script can verify SHA-256 and require GPG signature before executing binary; more approachable than teaching users about `npm config set` or `docker login`.
- Secondary deliverable: maintain Docker image for enterprise networks where binary download blocked; treat as optional path documented but not part of MVP acceptance checklist.
- Build pipeline detail:
  - Use bun 1.1.x locked via `.tool-versions` to ensure reproducible builds; run inside GitHub Actions ubuntu-22.04 + macOS-13 runners.
  - Post-build smoke tests: run `jasojeon-runner --version`, `jasojeon-runner pair --headless --dry-run`, `jasojeon-runner service template --print` to ensure binary integrity.
  - Store artifacts in S3 bucket `s3://releases.jasojeon.run/runner/<version>/`; CDN fronted by CloudFront with signed URLs to throttle downloads.
  - For first release, run limited beta with 20 volunteers; gather crash metrics before broad release.
  - Document fallback command `curl -O https://get.jasojeon.run/runner-vX.tar.gz && sh install.sh` for offline automation.

## 4. Pairing UX 개선

- Baseline (Stage 11.6): user runs `start-dev-runner.sh`, sees stdin prompt asking to paste pairing code shown on web; after input, runner polls backend, obtains device token, stores in secrets, reboots.
- Goal: move token exchange to browser to remove manual paste; minimize steps by using outbound-only channel.
- Candidate flows:
  1. Long-poll approach: runner exposes ephemeral claim id; backend instructs browser to POST token after user logs in; runner polls `/pairing/:claimId` every 2 seconds and auto-activates when token arrives; no localhost dependency; works behind firewalls.
  2. Localhost loopback: browser opens `http://127.0.0.1:<port>` after install to push token; near-instant but requires runner to expose HTTP server and instruct OS firewall to allow; fails when port blocked or user remote.
- Recommendation:
  - Use long-poll first: reuse existing start-dev-runner pairing handshake; CLI prints `Waiting for browser approval...` with spinner; browser orchestrates handshake via `pairing.jaeboong.shop` bridging view; once approved, runner receives token without user typing.
  - For UX, after curl install, script launches `jasojeon-runner pair --browser <url>` which opens default browser to login/pairing page containing QR + direct link; pair page uses WebSocket to signal success, so CLI can render large checkmark.
  - Provide fallback `--manual` flag to display code for copy/paste if browser not available (e.g., headless machine) while keeping Stage 11.6 path alive.
- UX instrumentation steps:
  - Browser pairing portal emits analytics events `pairing:start`, `pairing:token-issued`, `pairing:runner-confirmed`; cross-reference CLI logs to detect drop-off.
  - CLI prints colored states: "Waiting", "Approved", "Connected"; include timestamps for support.
  - Provide `--qr` flag to render ASCII QR linking to pairing page for kiosk scenarios.
  - Loopback fallback uses `127.0.0.1:54123` random port with CSRF token; only enabled when user passes `--loopback` flag because some security suites block local HTTP servers.
  - Document accessibility: ensure pairing page accessible with screen readers; allow alternative text-based code entry for visually impaired users.

## 5. Service Installation

- Approach: deliver `jasojeon-runner` binary plus `jasojeon-runner-supervisor.js` shim (thin wrapper around Stage 11.6 supervise.mjs) compiled via bun; services should not run as root.
- Linux (systemd user service):
  - Installer writes `~/.config/systemd/user/jasojeon-runner.service` referencing binary with args `run --config ~/.jasojeon/config.json`.
  - Service includes `Restart=on-failure`, `StartLimitIntervalSec=30`, `Environment=JASOJEON_ENV=prod`, `WorkingDirectory=~/.jasojeon`.
  - Installer runs `systemctl --user daemon-reload && systemctl --user enable --now ...`; instructs user to ensure `linger` enabled for unattended runs.
- macOS (launchd):
  - Generate `~/Library/LaunchAgents/com.jasojeon.runner.plist` with `ProgramArguments` pointing to binary; `KeepAlive` + `RunAtLoad` true.
  - Provide `launchctl bootstrap gui/$UID ...` command; ensure log path to `~/Library/Logs/JasojeonRunner.log`.
- Service wrapper responsibilities:
  - Manage Stage 11.6 supervise lifecycle: start runner, restart on exit, rotate logs, expose `jasojeon-runner status/logs` commands.
  - Offer `jasojeon-runner service install|uninstall|start|stop` subcommands to abstract OS differences; script calls these after binary drop.
  - Avoid OS-specific reimplementation by using config templates embedded in binary and rendered via moustache with user path/resolved binary location.
- Additional tasks:
  - Provide `jasojeon-runner service doctor` which checks whether service files exist, whether `systemctl --user status` is healthy, and whether launchd plist is loaded.
  - Document how to enable `loginctl enable-linger $USER` for Linux so runner persists after logout; include detection for servers missing `systemd --user` and fallback instructions using cron.
  - On macOS, instruct user to grant "Full Disk Access" only if logs need to read workspace files; default runner should require no special permissions beyond network.
  - Provide uninstall path verifying service is stopped before removing binary; handle failure to remove when service busy and prompt to rerun after 5 seconds.
  - Provide `jasojeon-runner logs --follow` command hooking into journalctl or tailing log file; ensures user does not need to know OS-specific commands.

## 6. Auto-update Channel

- Requirements: user devices must auto-update but allow defer/resume; update channel uses backend version API that Stage 11.6 runner polls at startup.
- Proposal:
  - Backend exposes `GET /api/runner/version?channel=stable` returning `{version, sha256, signature_url, release_notes}`.
  - Runner CLI checks once per hour (configurable) or when `jasojeon-runner update` invoked; downloads tarball to temp dir, verifies checksum, swaps symlink atomically.
  - Installer seeds default channel `stable`; allow `beta` via CLI flag for testers.
  - Use signature: sign manifest with Ed25519 key stored offline; CLI uses embedded public key; only after signature passes does it trust artifact.
  - Provide fallback offline update command: user downloads tarball manually and executes `jasojeon-runner install ./runner-vX.tar.gz`.
- Update choreography:
  - CLI writes `state/update.json` recording `currentVersion`, `targetVersion`, `downloadedAt`, `appliedAt`; supports support cases.
  - When update fails verification, CLI keeps previous binary, logs event, and surfaces actionable error referencing docs.
  - `jasojeon-runner update --check-only` prints latest version, release date, link to notes; helpful for verifying update channel health.
  - Add backend admin API to mark version as `deprecated` forcing CLI to upgrade before connecting; ensures security patches propagate.
  - Provide `--auto-update=false` flag for highly regulated users; highlight they accept risk; CLI still warns weekly if behind by >1 version.

## 8. Out of Scope

- Native Windows installers (MSI) and associated driver signing.
- Mobile platforms (iOS, Android) and tablet-specific flows.
- App Store, Mac App Store, or enterprise MDM distribution packaging.
- Multi-tenant device fleet management portal; limited to CLI-level operations.
- Hardware security modules or TPM-backed key storage; future stage.
- Runner UI front-end; Stage 11.6.B strictly CLI/service oriented.
- Non-x86/ARM64 architectures beyond mainstream (no RISC-V for MVP).
- Deferred concerns:
  - Offline-first organizations needing air-gapped updates; future stage may ship signed USB installers; for now, document manual steps but mark unsupported.
  - Multi-user machines with shared home directories; Stage 11.6.B assumes per-user install; multi-user support requires factoring service names.
  - Headless Linux servers without GUI; pairing flow still works via manual code entry; autop-run instructions added to docs but not optimized.

## 9. Risks

- macOS Gatekeeper + notarization: unsigned binaries blocked; need Developer ID cert + notarization pipeline; plan includes manual notarization initially, add automation later.
- Windows SmartScreen blocking general download; since Windows native excluded for MVP, document fallback (WSL instructions) but expect support load.
- Binary integrity: storing backend URL + device token locally; ensure encryption with OS keychain (macOS Keychain, Linux libsecret) or at least passphrase-protected secrets file; misuse may leak tokens.
- Token rotation: auto-update might restart service while token mid-rotation; need drain logic ensuring new binary waits for `stateHub` confirmation before dropping old connection.
- Network restrictions: outbound WSS might be MITM’d; need certificate pinning or at least CA bundle verification to avoid injection; fallback to HTTPS long-poll when WSS blocked.
- Support complexity: service misconfiguration may require manual `systemctl --user reset-failed`; include `jasojeon-runner doctor` command.
- Disk permissions: installer writes to `~/bin`; ensure the directory exists and is on PATH; add warning when PATH is missing.
- Telemetry privacy: ensure hashed machine id uses SHA-256 + namespace so it cannot be reversed; comply with policy; offer `--no-telemetry` flag.
- Build reproducibility: bun bundler updates might change binary size drastically; lock bun version in CI and document upgrade process.
- Dependency licensing: bundling Node modules requires verifying MIT/Apache compatibility; track in NOTICE file.
- Mitigation plans:
  - Acquire Apple Developer ID certificate before GA; run dry-run notarization each beta to ensure pipeline stable.
  - Provide `--token-path` override for air-gapped labs needing custom secure storage; default remains OS keychain.
  - Build integration tests simulating MITM to confirm TLS pinning logic rejects unexpected cert chains.
  - Document fallback when network proxies block WSS: CLI automatically downgrades to HTTPS long-poll but prints warning to user and admin dashboard.
  - Maintain `security.txt` contact for reporting binary tampering or supply-chain vulnerabilities.

## 10. Acceptance Criteria

- User on clean macOS or Ubuntu machine completes install via: `curl -fsSL https://get.jasojeon.run | bash`, signs in on browser, runner pairs and shows "Connected to jaeboong.shop" message, all within three discrete terminal interactions.
- After install, `jasojeon-runner status` reports paired device id, version, last heartbeat timestamp, and service is active; restart command works without re-pairing.
- Auto-update simulation: bump version on backend, runner detects update within 60 minutes (or via manual `update`), downloads new binary, restarts without dropping pairing token.
- Uninstall script removes binaries, services, and secrets (unless `--keep-secrets`); re-install reuses Stage 11.6 pairing flow seamlessly.
- `./scripts/apply-dev-stack.sh` integration: developer machines can still invoke dev stack and avoid interfering with user install artifacts because product plane deliverables live under `packages/runner/distribution`.
- QA sign-off: smoke test on macOS Sonoma + Ubuntu 22.04 verifying pairing, run start, abort, and log tail using only CLI commands.
- Documentation published: `docs/user/runner-installation.md` screenshot section includes Step 1/2/3 with consistent copy; support runbook updated.
- Verification script: QA writes `tools/verify-clean-machine.sh` to automate acceptance steps on ephemeral VM (macOS GitHub runner + Ubuntu cloud image) ensuring instructions accurate.
- Customer storyboards: design team prototypes install tutorial modal embedded inside jaeboong.shop guiding Step 1/2/3 with animations, ensuring frictionless user education.
- Support KPIs: success measured by install success rate >95% for first attempt and <10% tickets referencing Node/npm issues.
- Feature flagging: Stage 11.6.B tied to backend feature flag `runner_distributed_install`; we gate pairing portal entry until binary release ready to avoid half-baked flows.
- Release checklist: includes verifying version API returns correct TTL, CDN invalidations executed, docs updated, and fallback packages (Docker) built.
- Internal enablement session recorded and referenced by Support onboarding checklist; includes live pairing demo plus troubleshooting role-play.
- Observability dashboards (Grafana `runner-distribution` folder) track install attempts, pairing success rate, auto-update signals, and error codes; alarms wired into PagerDuty for degraded channels.
- Internal FAQ wiki enumerates known failure signatures (proxy blocks, Gatekeeper quarantine, missing `systemd --user`) and direct remediation scripts.
- Publish `docs/plans/2026-04-11-stage-11.6-b-runner-distribution.md` summary to company changelog to inform product + go-to-market stakeholders.
- Post-release retro scheduled (D+10) to capture metrics and backlog Stage 12 tasks such as Windows MSI, enterprise SSO integration, MDM policies.
