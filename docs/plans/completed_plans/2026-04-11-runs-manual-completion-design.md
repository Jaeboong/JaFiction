# Runs Manual Completion Design

**Goal:** 파이널라이저 라운드 종료와 세션 종료를 분리하고, 사용자가 명시적으로 문항을 완료/재개할 수 있게 실행 탭 UX와 세션 수명을 재설계한다.

## Context

현재 실행 탭에서는 파이널라이저가 한 라운드를 마치고 `run-completed` 흐름에 들어가면, 사용자 관점에서는 "라운드 종료"처럼 보이더라도 백엔드 활성 세션이 정리될 수 있다. 이 때문에 사용자는 같은 대화 맥락을 유지한 채로 추가 지시를 보내거나 문항 완료 여부를 직접 결정하기 어렵다.

사용자 의도는 다음과 같다.

- 파이널라이저는 라운드를 닫을 뿐, 세션을 닫으면 안 된다.
- 세션 종료는 사용자가 `중단`하거나 `문항 완료`를 눌렀을 때만 일어나야 한다.
- 완료 여부는 리뷰어 `APPROVE`와 무관하게 사용자가 결정한다.
- 완료 후에도 같은 위치에서 `재개`를 눌러 다시 활성화할 수 있어야 한다.

## Product Decision

### 1. 라운드 종료와 세션 종료를 분리한다

- 파이널라이저 완료는 `라운드 종료` 또는 `사이클 종료` 의미로만 취급한다.
- 이 시점에는 세션을 자동 정리하지 않는다.
- 사용자가 같은 실행에 대해 후속 메시지를 보내거나 다음 라운드를 이어갈 수 있어야 한다.

### 2. 세션 종료는 사용자 명시 액션으로만 일어난다

- `중단` 버튼을 누르면 세션이 종료된다.
- 새 `고정` 버튼을 누르면 해당 문항을 완료 처리하고 세션을 비활성화한다.
- 그 외 자동 흐름만으로는 세션을 닫지 않는다.

### 3. 완료 후 재개를 지원한다

- 완료 처리 후 헤더 액션의 `고정` 버튼은 같은 자리에서 `재개` 버튼으로 바뀐다.
- `재개`를 누르면 문항 completed 상태를 해제하고 다시 활성 상태로 전환한다.
- 사용자는 완료 이후에도 같은 문항 흐름을 다시 이어갈 수 있다.

## UI Direction

### Header actions

- 위치: `에이전트 대화` 카드 헤더 우측 액션 영역
- 배치: 실행 관련 버튼 왼쪽에 `고정` 의미의 완료 버튼을 둔다
- 이유:
  - 대화 본문이 길어져도 헤더는 항상 시야에 남는다
  - 세션 제어 액션이 이미 그 위치에 모여 있어 의미가 일관된다
  - 입력창이나 본문 하단에 둘 때보다 스크롤 의존성이 적다

### Visual states

- 활성 세션: 기존 활성 상태 색 체계를 유지한다
- 라운드 종료: 초록색은 라운드/사이클 종료 의미로만 유지한다
- 완료 처리: 회색 상태로 전환해 "사용자가 고정한 종료 상태"를 표현한다
- 완료 후 액션: 회색 상태에서 헤더 버튼은 `재개`로 노출된다

## Interaction Rules

### When the finalizer finishes a round

- 상태 라벨은 `라운드 종료` 또는 `사이클 종료`
- 세션은 유지
- 입력창은 계속 활성
- 사용자는 다음 중 하나를 선택
  - 추가 메시지 전송
  - 문항 완료
  - 중단

### When the user clicks complete/pin

- 문항 상태를 completed로 기록
- 세션을 비활성화
- 상태 표시를 회색 완료 상태로 전환
- 헤더 버튼을 `재개`로 교체

### When the user clicks resume

- completed 상태를 해제
- 해당 실행/문항을 다시 활성 상태로 되돌림
- 입력창과 개입 흐름을 다시 사용할 수 있게 함

## Architecture Direction

### Backend/session model

- 현재 `orchestrator.run()` 종료 시 무조건 세션 정리를 호출하는 흐름을 재검토한다
- 라운드 종료와 전체 종료를 별도 경로로 분리해야 한다
- 세션 정리는 다음 경우에만 수행한다
  - 사용자 중단
  - 사용자 문항 완료
  - 실패 같은 강제 종료 경로

### Completion state

- 기존 `essayAnswerStates` 기반 completed 모델을 재사용한다
- completed는 "사용자 확정 상태"로만 기록한다
- finalizer 출력만으로 completed 상태를 만들지 않는다

### Resume semantics

- `재개`는 completed를 해제하고 다시 개입 가능한 상태로 되돌리는 흐름이다
- 필요 시 기존 run을 활성 대상으로 복구하거나, 같은 문항 기반 continuation을 새로 생성하는 방식 중 현재 구조에 맞는 쪽을 구현 단계에서 선택한다

## Risks

- 현재 세션 수명 모델은 실행 스코프 종료와 강하게 묶여 있어, 라운드 종료 후 세션 유지에는 runner/session/orchestrator 정리가 필요할 수 있다
- completed 해제 후 어떤 run을 활성 대상으로 삼을지 정의가 필요하다
- 상태 색상과 액션 의미가 늘어나므로 `running / paused / round-complete / completed / aborted / failed`의 UX 정의를 한 번에 정리해야 한다

## Acceptance Criteria

- 파이널라이저가 끝나도 세션이 자동 종료되지 않는다
- 사용자는 라운드 종료 후에도 같은 대화 창에서 추가 입력을 보낼 수 있다
- 헤더 우측에 `고정` 완료 버튼이 보인다
- 완료 시 상태는 회색으로 바뀌고 입력은 비활성화된다
- 완료 후 같은 위치에 `재개` 버튼이 나타난다
- 재개를 누르면 다시 입력 가능 상태로 돌아온다
