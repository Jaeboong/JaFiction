# 프로젝트 컨텍스트 파일 업로드 복원 설계서

> ProjectsPage의 "컨텍스트 문서" 영역에서 PDF, MD, PPTX, TXT, 이미지 업로드를 다시 지원하도록 UI를 복원하는 설계

---

## 1. 문제 정의

현재 프로젝트 워크스페이스는 컨텍스트 문서를 두 방식으로 다뤄야 한다.

- 파일 기반 문서 업로드: PDF, PPTX, MD, TXT, 이미지 등
- 수동 텍스트 문서 작성

하지만 웹 UI에서는 현재 **수동 텍스트 추가만 노출**되어 있다.

- 프런트엔드: [`packages/web/src/pages/ProjectsPage.tsx`]에서 `수동 문서 추가` 폼만 존재
- 러너: [`packages/runner/src/routes/projectsRouter.ts`]에 `POST /api/projects/:projectSlug/documents/upload`가 이미 존재
- 저장소: [`packages/shared/src/core/storage.ts`]에 `importProjectUpload()`가 이미 존재

즉, 업로드 기능은 서버/스토리지에 살아 있는데, **웹 UI가 그 경로를 소비하지 않아서 사실상 사라진 것처럼 보이는 상태**다.

---

## 2. 목표

- 프로젝트 페이지에서 파일 업로드 기반 컨텍스트 추가를 다시 제공한다.
- PDF, MD, TXT, PPTX, 이미지 등 기존 `ContextExtractor`가 처리 가능한 파일을 그대로 허용한다.
- 수동 텍스트 추가는 유지하되, 업로드와 나란히 보이도록 정리한다.
- 기존 저장소 구조와 업로드 API는 최대한 재사용한다.

---

## 3. 범위와 가정

### 3.1 범위

- 대상 화면은 **ProjectsPage > ProjectWorkspace > 컨텍스트 문서 영역**
- 대상 문서는 **project scope documents**
- profile 문서 업로드 UX는 이번 설계 범위에 포함하지 않는다

### 3.2 가정

- 업로드 복원 요구는 "프로젝트 컨텍스트" 기준이다
- 백엔드 업로드 라우트는 현 상태를 기본 재사용한다
- 생성된 인사이트 문서는 일반 업로드 문서와 별개 UX로 다루며, 2번 설계 문서에서 분리 설계한다

---

## 4. 현재 상태 진단

### 4.1 프런트엔드

현재 ProjectsPage의 프로젝트 상세 화면은 아래 구조다.

- 프로젝트 기본 정보
- 컨텍스트 문서 목록
- 수동 문서 추가 폼

여기서 문제는 다음 두 가지다.

1. 업로드 진입점이 없다
2. 문서 관리 경험이 텍스트 작성 중심으로만 보인다

사용자 입장에서는 "컨텍스트 추가"가 곧 "텍스트 박스에 붙여넣기"처럼 보이며, 기존에 있던 파일 중심 워크플로우가 사라진다.

### 4.2 러너/스토리지

서버 쪽은 이미 준비돼 있다.

- `POST /api/projects/:projectSlug/documents/upload`
- `multer.memoryStorage()` 기반 멀티파일 업로드
- `storage.importProjectUpload(projectSlug, file.originalname, file.buffer)`
- 업로드 후 `stateStore.refreshProjects(projectSlug)`

즉 이번 작업의 핵심은 **새 기능 추가가 아니라 UI 복원 + 흐름 재정렬**이다.

## 5. 접근 옵션

### 옵션 A. 기존 페이지 안에 업로드 패널 복원

- "컨텍스트 문서" 섹션 아래에 업로드 패널 추가
- 수동 추가와 업로드를 같은 페이지 안에서 해결

장점

- 현재 구조와 가장 잘 맞는다
- 서버/API 변경이 거의 없다
- 사용자가 문서 목록과 추가 액션을 한눈에 본다

단점

- 페이지가 길어진다
- 수동 입력과 업로드가 한 화면에 같이 있어 밀도가 올라간다

### 옵션 B. 업로드 전용 모달 추가

- "파일 업로드" 버튼을 눌렀을 때 별도 모달에서 처리

장점

- 시각적으로 더 깔끔하다
- 드래그 앤 드롭, 업로드 진행 표시를 확장하기 좋다

단점

- 지금 코드베이스에 범용 모달 기반이 없다
- 업로드 복원만 하기엔 구조가 과해질 수 있다

### 옵션 C. 문서 관리 화면을 별도 워크스페이스로 분리

- 프로젝트 상세와 문서 관리를 분리

장점

- 장기적으로는 가장 확장성이 좋다

단점

- 현재 요구 범위를 크게 넘는다
- 정보 탐색 동선이 늘어난다

### 권장안

**옵션 A를 권장한다.**

이번 요구는 "사라진 업로드를 다시 살리는 것"이 우선이며, 기존 러너 경로가 이미 준비돼 있어서 **한 화면 내 복원형 접근이 가장 빠르고 자연스럽다.**

## 6. 제안 UX

### 6.1 프로젝트 화면 구조

`컨텍스트 문서` 영역을 아래처럼 재구성한다.

1. 컨텍스트 문서 목록
2. 파일 업로드 패널
3. 수동 문서 추가 패널

즉 "문서 목록" 아래에 "추가 방식 2종"이 이어지는 구조다.

### 6.2 파일 업로드 패널

패널 요소

- 버튼: `파일 업로드`
- 보조 문구: `PDF, PPTX, MD, TXT, 이미지 업로드 가능`
- 드래그 앤 드롭 영역 또는 최소한 숨겨진 `<input type="file" multiple>` 기반 선택
- 업로드 전 파일명 미리보기 리스트
- 업로드 실행 버튼

동작

- 여러 파일 동시 선택 가능
- 업로드 요청은 `multipart/form-data`
- 성공 시 문서 목록이 즉시 갱신됨
- 실패 시 업로드 실패 배너 또는 패널 하단 오류 메시지 노출

### 6.3 수동 문서 추가 패널

기존 폼은 유지하되 이름을 더 명확히 한다.

- 섹션명: `텍스트 문서 직접 추가`
- 필드: 제목, 내용
- 선택 필드로 `노트` 추가 가능

### 6.4 문서 목록 표시

목록은 기존 카드 패턴을 유지하되 다음을 분명히 보여준다.

- 파일명/제목
- 문서 타입 배지 (`PDF`, `PPTX`, `Markdown`, `이미지` 등)
- 정규화 상태
- 기본 포함 여부
- 생성 시각

## 7. API 및 데이터 흐름

### 7.1 재사용 API

기존 엔드포인트를 그대로 사용한다.

`POST /api/projects/:projectSlug/documents/upload`

Request

- `multipart/form-data`
- field: `files`

Response

- 현재 구현대로 전체 문서 목록 반환

### 7.2 클라이언트 추가 메서드

`RunnerClient`에 아래 메서드 추가를 설계한다.

```ts
uploadProjectDocuments(projectSlug: string, files: File[]): Promise<void>
```

주의점

- 현재 `RunnerClient.request()`는 JSON 요청 중심이라 업로드 전용 fetch 경로를 별도 두는 편이 낫다
- `Content-Type`는 브라우저가 `FormData`에 맞게 자동 설정하도록 수동 지정하지 않는다

### 7.3 상태 갱신

업로드 성공 후 갱신 방식은 두 경로가 동시에 작동한다.

- 서버가 `refreshProjects(projectSlug)` 수행
- 프런트는 기존 `/ws/state` 수신으로 최신 상태 반영

즉 클라이언트가 별도 재조회하지 않아도 된다.

## 8. 화면 상세 설계

### 8.1 권장 레이아웃

```text
컨텍스트 문서
├─ 문서 카드 목록
├─ 파일 업로드
│  ├─ 파일 선택 / 드롭
│  ├─ 선택 파일 리스트
│  └─ 업로드 실행
└─ 텍스트 문서 직접 추가
   ├─ 제목
   ├─ 내용
   └─ 저장
```

### 8.2 상태 설계

업로드 패널 로컬 상태

- `selectedFiles: File[]`
- `uploadStatus: idle | pending | failed`
- `uploadError?: string`

수동 입력 상태와 분리한다.

이유

- 텍스트 작성 중 업로드를 해도 폼 내용이 날아가지 않아야 한다
- 두 입력 수단은 독립 워크플로우다

---

## 9. 에러 처리

대표 시나리오

- 파일이 0개 선택됨
- 지원하지 않는 파일 형식
- 추출 실패
- 일부 파일만 실패

이번 단계 권장 처리

- 1차 설계에서는 현재 서버의 일괄 업로드 응답 구조를 그대로 사용
- 에러는 전체 요청 실패로 처리
- 추후 고도화 시 파일별 성공/실패 리포트 응답으로 확장 가능

---

## 10. 테스트 관점

### 10.1 UI 테스트

- 파일 선택 버튼 클릭 후 파일명 리스트가 보인다
- 업로드 성공 시 문서 목록에 새 카드가 추가된다
- 업로드 실패 시 오류 메시지가 보인다

### 10.2 통합 테스트

- `documents/upload` 요청이 실제 multipart로 전송된다
- 업로드 후 websocket 상태 갱신으로 프로젝트 문서 목록이 업데이트된다

### 10.3 회귀 포인트

- 기존 수동 텍스트 추가가 그대로 동작해야 한다
- 생성된 프로젝트 문서가 RunsPage의 selectedDocumentIds 흐름에 그대로 포함돼야 한다

## 12. 결론

이 요구의 본질은 새로운 시스템 도입이 아니라, **이미 있는 업로드 기능을 프로젝트 화면에서 다시 보이게 만들고 문서 추가 경험을 정상화하는 것**이다.

권장 방향은 다음과 같다.

- 프로젝트 페이지 안에서 파일 업로드 패널 복원
- 수동 텍스트 입력과 병렬 제공
- 기존 업로드 라우트/스토리지 재사용
- 문서 목록은 그대로 프로젝트 컨텍스트의 단일 진입점 유지

이 설계는 구현량 대비 복원 효과가 크고, 이후 문서 편집/미리보기 고도화로도 자연스럽게 이어진다.
