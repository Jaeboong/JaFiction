# Stage 11.6 — Hosted dev/prod boot automation

**Date**: 2026-04-11
**Branch**: `feat/hosted-migration`
**Depends on**: 11.7 (local mode retired), 11.8 (profile parity restored)

## Why

After 11.7 the runner no longer exposes a localhost REST/WS server. The hosted
dev loop requires five moving parts to work in concert:

1. Postgres
2. Redis
3. Backend (Fastify, port 4000)
4. Runner (outbound WS client against backend)
5. Web dev server (Vite, port 4124)

Today there is no single command to bring this up. `scripts/dev-stack.sh` (from
11.7) requires `JASOJEON_BACKEND_URL` to already be exported, assumes the
backend is already running, and assumes the runner has already been paired. A
developer (or the author dogfooding the service) has to run ~5 separate
commands in the right order, manually handle first-run pairing, and restart
crashed processes by hand.

This is also the foundation for eventual end-user runner distribution: the
runner bootstrap script (pairing + supervised run) is the same flow a real user
would follow, just pointed at `https://jaeboong.shop` instead of
`http://localhost:4000`. Build it once here, reuse for Stage 11.6.B.

## Goal

`./scripts/dev-stack.sh` with **zero required env vars** brings up the full
hosted dev loop from a clean checkout (modulo `.env.dev` being populated
once). First run guides the user through pairing. Subsequent runs are silent.

## Scope

### Part A — this stage (dev ergonomics)

Everything local, one command up, one command down, crash recovery.

### Part B — out of scope (flagged, not built)

- Packaged runner binary (pkg / bun build / single-file exe)
- systemd/launchd service installer
- Auto-update
- GitHub Actions CI runner image

## Deliverables

### 1. Dev infrastructure compose — `docker-compose.dev.yml` (new)

Containerized **postgres + redis only**. Backend and runner run on the host
via `tsx` so edit loops stay fast — no image rebuild between changes.

- Postgres 16 alpine, published on `localhost:5433` (avoid clash with a
  possible native postgres on 5432). Volume: `jasojeon_dev_pg`.
- Redis 7 alpine, published on `localhost:6380` (avoid clash). Volume:
  `jasojeon_dev_redis`.
- Healthchecks identical to production compose.
- No nginx, no backend image.

### 2. Dev env template — `packages/backend/.env.dev.example` (new)

Committed template. Copy-to-`.env.dev` on first run. Defaults aligned with the
dev compose:

```
DATABASE_URL=postgres://jasojeon:devpass@localhost:5433/jasojeon
REDIS_URL=redis://localhost:6380
GOOGLE_CLIENT_ID=<fill me in>
GOOGLE_CLIENT_SECRET=<fill me in>
COOKIE_SECRET=<openssl rand -hex 32>
PORT=4000
NODE_ENV=development
PUBLIC_BASE_URL=http://localhost:4000
```

`.env.dev` itself must be in `.gitignore`.

### 3. Shared helper — `scripts/lib/supervise.mjs` (new)

Tiny Node supervisor. Spawns a child process, restarts it on non-zero exit
with exponential backoff (1s, 2s, 4s, 8s, cap 30s). Gives up after 5
consecutive failures in under 30s and exits non-zero. Writes pid of the
supervisor (not the child) to the pid file so stop scripts have a stable
handle. Forwards SIGINT/SIGTERM to the child and exits cleanly.

CLI: `node supervise.mjs --label runner --pidfile ... --logfile ... -- <cmd> <args...>`

### 4. Backend dev starter — `scripts/start-dev-backend.sh` (new)

Flow:
1. Ensure `packages/backend/.env.dev` exists. If missing, copy the example and
   fail loudly with next-step instructions.
2. Ensure `docker-compose.dev.yml` is up (`docker compose -f docker-compose.dev.yml up -d`).
   Wait for both healthchecks green (60s timeout).
3. Run `npm run db:migrate -w packages/backend` with `.env.dev` loaded.
4. Start backend under `supervise.mjs`: `tsx packages/backend/src/index.ts`
   (no `watch` — supervisor handles restart; tsx watch + supervisor overlap
   causes double restarts).
5. `wait_for_http http://localhost:4000/healthz 30`.
6. PID file: `.harness/pids/backend.pid`. Log: `.harness/logs/backend.log`.

### 5. Runner bootstrap — `scripts/start-dev-runner.sh` (new) — **the reusable part**

Takes optional `--backend-url` (default from env or `http://localhost:4000`).

Flow:
1. Call a small helper — new `packages/runner/src/hosted/checkToken.ts` CLI
   that exits 0 if `loadDeviceToken()` returns a value, 1 otherwise. Invoked
   via `tsx`.
2. **If no token**:
   - Print: `→ Open ${backendUrl} in your browser, sign in, and go to Settings → Devices → Add device.`
   - Print: `→ Paste the 8-character pairing code here: `
   - Read code from stdin (support `$JASOJEON_PAIRING_CODE` env to skip
     prompt, for non-interactive cases).
   - Run runner once as `JASOJEON_MODE=pair` with the code. If pairing fails,
     print the error and exit 1.
   - On success, print "Paired. Starting runner..." and fall through.
3. Start runner under `supervise.mjs`: `tsx packages/runner/src/index.ts` with
   `JASOJEON_MODE=hosted`, `JASOJEON_BACKEND_URL=<url>`.
4. Tail the log until it contains `hosted mode — connecting to` (30s timeout).
5. PID file: `.harness/pids/runner.pid`. Log: `.harness/logs/runner.log`.

**Why this is reusable**: swap `http://localhost:4000` for
`https://jaeboong.shop`, drop the supervisor wrapper into a systemd unit or
launchd plist in 11.6.B, and the same script is the end-user install path.

### 6. `scripts/dev-stack.sh` — rewrite

No required env vars. Sequence:
1. `./scripts/check.sh` (gate bad commits).
2. `./scripts/stop-dev-stack.sh` (idempotent clean slate).
3. `./scripts/start-dev-backend.sh`.
4. `./scripts/start-dev-runner.sh` (defaults to `http://localhost:4000`).
5. `./scripts/start-dev-web.sh`.
6. `./scripts/status-dev-stack.sh`.

Support flags: `--no-infra`, `--no-backend`, `--no-runner`, `--no-web`,
`--skip-check`. All off by default.

### 7. `scripts/stop-dev-stack.sh` — extend

Stop web, runner (supervisor pid → supervisor forwards SIGTERM to child),
backend. Add `--all` flag that also runs
`docker compose -f docker-compose.dev.yml down`. Default behavior: keep
containers running (faster restarts).

### 8. `scripts/status-dev-stack.sh` — extend

Show per-component status:
- Postgres container: up/down + port
- Redis container: up/down + port
- Backend: pid + `/healthz` OK
- Runner: pid + whether log contains recent `hosted mode — connecting`
- Web: pid + `http://localhost:4124` OK

### 9. Backend health endpoint

Already exists at `/healthz`. No change needed. If the implementer finds it
unavailable, add a trivial `{ok: true}` handler.

### 10. Docs updates

- `CLAUDE.md` — entrypoint table: `dev-stack.sh` is the canonical dev loop,
  `apply-dev-stack.sh` marked as web-only convenience (it still exists).
- `docs/development/OPERATING_RULES.md` — add a short section on the dev
  automation: "one command up, one command down, first-run pairing".
- `README.md` (if present) — quickstart pointing at `dev-stack.sh`.

## Acceptance criteria

1. Clean-slate flow (no `.env.dev`, no token): `./scripts/dev-stack.sh` prints
   clear next-step instructions when `.env.dev` is missing, and after the user
   creates it, brings the stack up and walks them through pairing
   interactively.
2. Warm flow (token exists, `.env.dev` exists): `./scripts/dev-stack.sh` with
   no args brings everything up in under 60 seconds, end to end, with no user
   interaction.
3. Killing the runner process manually (`kill <pid>`) causes the supervisor to
   restart it within 5 seconds. Same for backend.
4. `./scripts/stop-dev-stack.sh` stops web + runner + backend cleanly. `--all`
   also stops containers.
5. `./scripts/status-dev-stack.sh` shows all 5 components with correct
   up/down state.
6. `./scripts/check.sh` still passes.
7. A second `dev-stack.sh` invocation after `stop-dev-stack.sh` is idempotent
   and succeeds.

## Out of scope (Stage 11.6.B)

- Single-file runner binary.
- systemd/launchd service unit generation.
- Auto-update channel.
- Long-poll pairing UX (backend endpoint change).
- Bundling runner + web for end users without Node installed.

## Judgment latitude for implementer

- If `tsx watch` fights with the supervisor, use plain `tsx` under supervisor.
- If dev docker compose port collisions are detected, pick the next available
  pair and write them into `.env.dev.example`.
- If `/healthz` gating blocks because of missing Google OAuth env (backend
  might refuse to start without valid GOOGLE_CLIENT_ID), either (a) allow
  placeholder values that fail closed on `/auth/*` routes but allow `/healthz`
  to succeed, or (b) document the requirement in the `.env.dev.example`
  header. Prefer (b).
- If interactive stdin prompting on WSL behaves poorly, fall back to
  `$JASOJEON_PAIRING_CODE` env and print clear instructions.
- If the profile secrets store (used for device token) has a lock contention
  issue when the supervisor restarts the child rapidly, add a 500ms grace
  delay in the restart loop.

## Risk notes

- Runner supervisor + file secret store: concurrent access during rapid
  restarts could corrupt the lock. Test the restart path explicitly.
- Existing live dev stack is still running the pre-11.7 runner binary.
  Implementer must NOT run `apply-dev-stack.sh`. Only `check.sh` for
  validation.
- `.env.dev.example` must NOT contain real secrets. Use placeholders and
  document generation in the header.
